
# 一、推荐的三人分工
我建议不要按“Lex 一个人、Yacc 一个人、语义一个人”完全切死，而是采用：
* **A：词法主负责人**
* **B：语法主负责人**
* **C：语义与系统集成主负责人**

⠀同时每个人都承担一点测试 / 文档 / 答辩材料，这样更稳。

### 核心职责
负责 .l 文件相关的完整链路：
* Lex 输入文件结构解析
* 正则表达式解析
* 正则转 NFA
* NFA 合并
* DFA 确定化
* DFA 最小化
* 根据 DFA 实现词法分析器
* token 输出格式设计，与 Yacc 模块对接

⠀对应老师计划
这部分直接对应老师 PPT 里的：
* “确定 Lex 输入文件”
* “Lex 输入文件的解析”
* “Lex RE => 常规 RE”

* “将所有正规式对应的 NFA 合并为一个大的 NFA”

# 二、三人之间的接口约定
这个特别重要，建议第 7 周就定死，不然后面会反复返工。
# A → B 接口
词法分析输出给语法分析的 token 统一格式：
### (token_type, lexeme, line, column, attr)
例如：
### (ID, "sum", 3, 5, symbol_id=12)
### (NUM, "123", 3, 9, value=123)
### (PLUS, "+", 3, 13, null)
这样 B 做语法分析时不需要关心词法内部实现。
# B → C 接口
语法分析向语义模块提供两种东西：
* 规约序列 / 产生式序列
* 语法树 / 带语义动作节点的语法树

⠀建议你们优先统一成：
* **主接口：语法树**
* **辅接口：规约日志**

⠀因为老师拓展 PPT 里已经强调了“带语义动作节点的语法树”“先序遍历语法树执行语义动作”。
# C → 全组接口
C 负责定义这些公共数据结构：
* Token
* SymbolEntry
* TypeExpr
* ASTNode
* Quadruple

# 八、最稳的实现策略
你们最好不要一开始就冲 c99 全集。更稳的做法是：
1. **先拿** minic **或自己裁剪过的子集跑通全流程**
2. 再逐步扩展声明、函数、控制流、类型检查
3. Lex/Yacc 可以支持较大集合
4. **中间代码 / 类型检查只做语言子集**，这个其实和老师拓展 PPT 里的建议完全一致： “Lex、Yacc部分针对全集，中间代码部分针对子集。”

⠀这能大大降低你们翻车概率。
